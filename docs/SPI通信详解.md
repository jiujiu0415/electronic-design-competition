# STM32 SPI 通信详解（大白话版）

> 写给大一同学——先理解，再写代码。

---

## 一、SPI 是什么？用人话解释

假设你在教室里，老师想让你去黑板写答案。怎么做？

1. 老师喊你名字 → 你站起来（**选中设备**）
2. 老师念题目 → 你听着（**数据发过去**）
3. 你说答案 → 老师记下来（**数据回过来**）
4. 老师让你坐下 → 下一个（**释放设备**）

**SPI 就是芯片之间这样"一问一答"的方式。**

```
        老师（主设备）          学生（从设备）
    ┌──────────────┐       ┌──────────────┐
    │  喊名字（CS） │──────▶│   被选中      │
    │  念题目（MOSI）│──────▶│   听题目      │
    │  记答案（MISO）│◀──────│   说答案      │
    │  打节拍（SCK） │──◀──▶│   跟着节奏    │
    └──────────────┘       └──────────────┘
```

在 STM32 + AD9833 这个项目里：
- **STM32 是老师**（主设备 Master）
- **AD9833 是学生**（从设备 Slave）

## 二、4 根线的分工

| 线名 | 全称 | 作用 | 谁控制 |
|------|------|------|--------|
| **SCK** | Serial Clock | 时钟线——打节拍，告诉对方"现在可以读了" | Master |
| **MOSI** | Master Out Slave In | 主设备发数据给从设备 | Master 发 |
| **MISO** | Master In Slave Out | 从设备回数据给主设备 | Slave 回 |
| **CS/FSYNC** | Chip Select | 选中某个从设备（低电平有效） | Master |

### 关键理解：

```
SCK（时钟）就像节拍器：

没有节拍：  数据线上的电压没有意义
节拍响了：  "现在！读这根线上的值！"

         ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐
SCK:  ───┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─
         ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑
        读  读  读  读  读  读  读  读
                          
MOSI: ───1───0───1───1───0───0───1───0───  → 发了 10110010 (8个bit)
```

## 三、数据怎么传输的？全双工交换

SPI 最妙的地方：**一边发一边收，同时进行。**

```
想象两个人在玩传送带：

        Master                              Slave
      ┌────────┐                        ┌────────┐
      │  [7]   │ ────MOSI────▶          │  [?]   │
      │  [6]   │                        │  [?]   │
      │  [5]   │                        │  [?]   │
      │  [4]   │                        │  [?]   │
      │  [3]   │                        │  [?]   │
      │  [2]   │                        │  [?]   │
      │  [1]   │                        │  [?]   │
      │  [0]   │                        │  [?]   │
      │        │ ◀────MISO────          │  [7]   │
      │  [?]   │                        │  [6]   │
      │  [?]   │                        │  [5]   │
      │  [?]   │                        │  [4]   │
      │  [?]   │                        │  [3]   │
      │  [?]   │                        │  [2]   │
      │  [?]   │                        │  [1]   │
      │  [?]   │                        │  [0]   │
      └────────┘                        └────────┘

每次钟响，左边进一个，右边出一个。8次钟响，交换完了。
```

**这就是"全双工"**——你用 Master 的一个字节，换来了 Slave 的一个字节。

## 四、SPI 的四种"节奏"（模式）

不是所有芯片在同一个节奏上工作。SPI 有 4 种模式，区别在于两个参数：

### CPOL（时钟空闲时是高还是低）
- **CPOL=0**：没事的时候，时钟线是 **低电平**
- **CPOL=1**：没事的时候，时钟线是 **高电平**

### CPHA（在哪个边沿读数据）
- **CPHA=0**：第 **1** 个边沿读（上升沿 或 下降沿 取决于 CPOL）
- **CPHA=1**：第 **2** 个边沿读

### 四种模式速查：

| 模式 | CPOL | CPHA | 什么时候采样？ |
|------|------|------|---------------|
| Mode 0 | 0 | 0 | 上升沿读 |
| Mode 1 | 0 | 1 | 下降沿读 |
| **Mode 2** | **1** | **0** | **下降沿读** ← AD9833 用这个 |
| Mode 3 | 1 | 1 | 上升沿读 |

### 为什么 AD9833 用 Mode 2？

```
AD9833 手册里写的是：
  "Data is clocked into the AD9833 on the falling edge of SCLK."

翻译：数据在 SCLK 的下降沿被写入 AD9833。

时钟空闲时 = 高电平 → CPOL=1
第1个边沿（下降沿）读 → CPHA=0
所以是 Mode 2 (CPOL=1, CPHA=0)
```

图示：

```
SCLK (CPOL=1)
  ─────┐   ┌───┐   ┌───┐   ┌───┐   空闲时高电平
       │   │   │   │   │   │   │
       └───┘   └───┘   └───┘   └───
        ↑       ↑       ↑       ↑
     下降沿   下降沿   下降沿   下降沿
     (AD9833 在这些时刻采集数据)
     
SDATA
  ──────────────────────────────────  数据在这个时刻必须稳定
       ↑       ↑       ↑       ↑
     bit15   bit14   bit13   bit12  ...
```

## 五、FSYNC（片选）的特殊之处

一般的 SPI 有 CS（Chip Select），传输期间保持低电平就行。

**但 AD9833 的 FSYNC 不一样**——它要求在每 16-bit 传输完成后拉高，意思是"这一帧结束了，可以执行了"。

```
通常 SPI:
  CS:  ────────┐                ┌──────  拉低期间传输任意多字节
               └────────────────┘

AD9833:
  FSYNC:  ──┐  ┌──┐  ┌──┐  ┌──  每16bit拉高一次
             └──┘  └──┘  └──┘
         ┌──┬──┐ ┌──┬──┐ ┌──┬──┐
  SDATA: │16│bit│ │16│bit│ │16│bit│  ...  一帧=16bit
         └──┴──┘ └──┴──┘ └──┴──┘
```

**这就是为什么我们不能用 STM32 的硬件 NSS（自动片选），必须用 GPIO 软件控制 FSYNC。**

## 六、在 STM32CubeIDE 里怎么配 SPI？

以 STM32F103C8T6 + AD9833 为例：

### 步骤 1：打开 CubeMX，选 SPI1

```
Pinout view:
  PA5 → SPI1_SCK   (自动分配)
  PA7 → SPI1_MOSI  (自动分配)
  PA4 → GPIO_Output (手动设为推挽输出，做 FSYNC)
```

### 步骤 2：配置 SPI1 参数

```
Mode:                Full-Duplex Master
Data Size:           16 Bit          ← AD9833 命令都是16位的
Baud Rate Prescaler: 256             ← 72MHz/256 ≈ 281kHz (AD9833 最高40MHz，够用)
CPOL:                High            ← 空闲时高电平
CPHA:                2 Edge          ← 第二个边沿 = 第一个下降沿
NSS:                 Software        ← 我们用 GPIO 代替
First Bit:           MSB First       ← 高位先发
```

怎么理解 Baud Rate Prescaler？

```
SPI 时钟 = 72MHz ÷ 分频系数

72MHz ÷ 2   = 36MHz   → 太快，可能不稳定
72MHz ÷ 4   = 18MHz   → 
72MHz ÷ 8   = 9MHz    →
72MHz ÷ 16  = 4.5MHz  →
72MHz ÷ 32  = 2.25MHz →
72MHz ÷ 64  = 1.125MHz→
72MHz ÷ 128 = 562.5kHz→
72MHz ÷ 256 = 281.25kHz → 稳！（推荐）
```

**AD9833 最高支持 40MHz SCLK，我们用 281kHz 虽然慢但绝对稳定，先跑通再说。**

## 七、代码怎么写？（HAL 库）

SPI 通信只需要 3 个 HAL 函数：

### 1. 发数据（AD9833 只接收，MISO 这条线在本项目中不接）

```c
uint16_t data = 0x2100;  // 要发的16位数据

HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET);  // 拉低 FSYNC（开始）
HAL_SPI_Transmit(&hspi1, (uint8_t *)&data, 1, 100);     // 发送16位
HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);    // 拉高 FSYNC（结束）
```

### 2. 如果设备有回传数据，用 TransmitReceive

```c
uint16_t tx_data = 0x0000;  // 发过去的（可能是dummy）
uint16_t rx_data = 0;       // 收回来

HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET);
HAL_SPI_TransmitReceive(&hspi1, (uint8_t *)&tx_data, (uint8_t *)&rx_data, 1, 100);
HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);
// rx_data 里就是从设备读回来的值
```

### 3. 完整流程：每次操作 FSYNC → Transmit → FSYNC

```c
/* 把这个流程封装成一个函数 */
void AD9833_Write(uint16_t data) {
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_RESET);   // ①片选拉低
    HAL_SPI_Transmit(&hspi1, (uint8_t *)&data, 1, 100);      // ②发16bit
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_4, GPIO_PIN_SET);      // ③片选拉高
}
```

## 九、常见问题

**Q: 为什么我的 PA4 (FSYNC) 不用 SPI 的 NSS 引脚？**

因为 AD9833 需要在每 16-bit 后拉高 FSYNC 来"锁存命令"。STM32 硬件 NSS 是传输期间一直低，不符合 AD9833 的要求。所以用 GPIO 手动控制。

**Q: MISO 线要不要接？**

本项目不需要。AD9833 是一个"只写"设备——你只能给它发命令，它不会回传数据。

**Q: 281kHz 会不会太慢？**

够用。AD9833 一个命令 16 bit，281kHz 下约 57μs 完成一次传输，更新频率完全够快。

**Q: 怎么知道我真的发出去了？**

用逻辑分析仪或示波器看 SCK 和 MOSI 的波形。没有仪器的话，先确保代码编译通过，等后面接上 AD9833 看有没有波形输出就能验证。
