## 一、问题分析框架

### 1.1 用户画像评估
面对一个新备赛者，首先需要厘清：
- **年级/基础**：决定内容深度（大一 → 从零开始，不假设先修知识）
- **竞赛经验**：决定策略侧重点（首次 → 以体验为主，长期规划）
- **技术栈**：决定工具推荐方向（已有 STM32/MSP430 基础）
- **时间窗口**：决定学习节奏（13个月，时间充裕）

### 1.2 需求矩阵
将用户模糊需求转化为可执行方向：

| 需求维度 | 具体化 |
|---------|--------|
| 硬件设计 | 电路原理、PCB、元器件选型 |
| 编程/嵌入式 | MCU编程、外设驱动、算法 |
| 备赛规划 | 时间线、学习路线、团队 |
| 赛题分析 | 历年真题、出题趋势 |

## 二、平台选型决策

### 决策原则
1. **生态优先**：资料多 > 性能强（对新手而言）
2. **竞赛主流**：选择大多数参赛队用的平台
3. **渐进式**：先精通一个，再扩展

### 最终选择：STM32F1/F4 主攻

```
STM32  → 生态最好，教程最多，HAL库友好
MSP430 → 低功耗专项，按需突击
FPGA   → 大一暂不推荐，入门门槛高
```

## 三、学习路线设计方法

### 设计原则
- **阶段性**：每阶段有明确的输入和输出
- **可检验**：每阶段结束时能做具体的事
- **螺旋上升**：基础→模块→整合→实战

### 13个月路线图
```
月份    阶段              核心产出
1-2     打基础            C语言+电路+STM32入门+仪器
3-5     核心技能          模电/数电+STM32进阶+简单模块
6-8     整合应用          传感器+电机+显示+通信+小项目
9-11    真题实战          分类专项训练+模块库+速度训练
12-13   冲刺             模拟赛+模板整理+团队磨合
```

## 四、技能扩展方法论

### 4.1 发现可用资源
使用 `npx skills find <关键词>` 系统性搜索，关键词应覆盖：
- 硬件层：stm32, embedded, baremetal, gpio, pwm, adc, spi, i2c, uart, dma
- 电路层：circuit, pcb, schematic, spice, kicad, eda
- 信号层：signal processing, filter, sensor
- 系统层：freertos, rtos, firmware, power
- 供应链：bom, datasheet, battery, jlcpcb, lcsc

### 4.2 评估标准
| 指标 | 阈值 | 说明 |
|-----|------|------|
| 安装量 | >1K 优先，<100 谨慎 | 社区验证 |
| 来源 | 官方组织优先 | vercel-labs, anthropics, matlab |
| 匹配度 | 直接覆盖备赛需求 | 不追求泛用工具 |
| 维护状态 | 近期有更新 | 避免死项目 |

### 4.3 安装策略
```
首选：npx skills add <repo@skill> -g -y
备选：git clone + 手动复制 → ~/.claude/skills/
网络受限：SSH协议替代HTTPS
```

## 五、网络问题处理

### 遇到的问题
- `github.com:443` 直连不稳定（国内网络环境）
- `ghproxy.com` 等镜像同样不通
- `npx skills add` 底层 HTTPS clone 失败

### 解决方案
```
HTTPS → SSH 协议切换
git clone git@github.com:owner/repo.git
```

### 通用处理流程
1. 先用 `npx skills find` 搜索（npm registry 通常可达）
2. 再尝试 `npx skills add` 安装
3. 失败时用 `git clone git@github.com:...` SSH 克隆
4. 手动 `cp -r` 到 `~/.claude/skills/`
